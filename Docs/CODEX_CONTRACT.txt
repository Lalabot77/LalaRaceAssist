# LALALAUNCH CODEX CONTRACT

This contract applies to ALL Codex tasks unless explicitly overridden.

Codex must treat this as mandatory engineering policy.

---

## 1. Scope Discipline (No Feature Creep)

- Modify ONLY the files explicitly listed in the task, plus mandatory documentation files required to keep repo truth aligned.
- Do NOT expand scope.
- Do NOT introduce new subsystems.
- Do NOT add new UI controls unless explicitly requested.
- Do NOT change learning, shift logic, fuel math, CarSA logic, or telemetry gating unless explicitly requested.

If something appears wrong but is outside scope:
- Do not fix it.
- Do not refactor it.
- Leave it untouched.

---

## 2. Documentation Is Mandatory

Every code change must update relevant documentation.

Always evaluate and update as needed:

Technical / repo docs:
- `Docs/SimHubParameterInventory.md` (exports / settings / properties)
- `Docs/SimHubLogMessages.md` (new or modified log lines)
- `Docs/Subsystems/<affected_subsystem>.md`
- `Docs/Plugin_UI_Tooltips.md` (if UI text changes)
- `Docs/Project_Index.md` (if subsystem description, navigation, or canonical doc map changes)
- `Docs/RepoStatus.md` (update validation commit + summary)

GitHub-facing user docs:
- `README.md`
- `CHANGELOG.md`
- `Docs/Quick_Start.md`
- `Docs/User_Guide.md`
- any relevant user-facing feature page such as:
  - `Docs/Dashboards.md`
  - `Docs/Strategy_System.md`
  - `Docs/H2H_System.md`
  - `Docs/Rejoin_And_Pit_Assists.md`
  - or other equivalent user docs that own the changed feature

Rules:
- If a task changes user-facing behaviour, workflow, UI labels, tab names, settings location, dash behaviour, install steps, or practical usage guidance, GitHub-facing user docs must be reviewed and updated in the same task.
- If a task changes only internal implementation with no user-visible effect, user docs may remain unchanged, but Codex must explicitly state that no user-facing documentation changes were required.
- Documentation must reflect final behavior.
- Remove outdated statements.
- Do not mention deprecated settings unless marked `legacy (unused)`.
- Do not document inactive systems as active features.

No task is complete without documentation alignment.

---

## 3. Codex Execution Method (Mandatory Working Order)

For every non-trivial task:

- Read `Docs/Project_Index.md` first.
- Use `Docs/Project_Index.md` to locate the canonical docs for the affected area.
- Read the relevant `Docs/Subsystems/*.md` files before editing code or subsystem docs.
- Check `Docs/RepoStatus.md` before implementation when current validated state matters.
- Treat `Docs/Code_Snapshot.md` as orientation only. If it conflicts with canonical docs, the canonical docs win.

Before editing on any non-trivial task:

- Do a no-edit analysis pass first.
- State the relevant code paths and documentation paths being touched.
- State the system invariants that must remain true.
- State the edit boundaries and keep them tight.
- Explicitly state whether GitHub-facing user docs are expected to change.

While implementing:

- Prefer subsystem-local documentation updates over growing central/global docs.
- Keep edits inside the owning subsystem where practical.
- Do not widen file responsibilities without a clear need.
- Keep architecture guardrails in mind; extend existing subsystem seams before creating new cross-cutting behaviour.
- If canonical docs are missing or unclear, update the canonical doc rather than relying on ad hoc task wording.
- If user-facing behaviour changes, update the owning GitHub-facing docs in the same task instead of leaving it for later.

At task close:

- Re-check the edited docs/files against the final repo state.
- Update `Docs/RepoStatus.md` with the validation commit and summary when documentation or behaviour changed.
- Confirm whether `README.md`, `CHANGELOG.md`, and user guides were reviewed and whether they changed.

---

## 4. Stability First

This project is actively under test.

Therefore:

- Do NOT alter runtime behavior outside task scope.
- Do NOT modify `ShiftAssistEngine` logic unless explicitly requested.
- Do NOT change learning behavior unless explicitly requested.
- Do NOT introduce timing side effects in telemetry loop.
- Do NOT block main thread.
- Do NOT introduce race conditions.

---

## 5. C# and Dependency Rules

- Target C# 7.3 compatibility.
- No new NuGet packages unless explicitly requested.
- No new external DLLs.
- No unsafe code.
- No async/await changes unless explicitly requested.
- No WASAPI exclusive mode unless explicitly requested.

---

## 6. Audio System Rules (Shift Assist)

Unless explicitly overridden:

- Primary beep behavior must remain unchanged.
- Urgent beep must not modify learning, exports, or shift target logic.
- Volume overrides must use existing scaling/cache logic.
- No duplication of audio backend state.

---

## 7. Logging Discipline

- No log spam.
- Warnings must be one-time where possible.
- Keep existing log format conventions.
- Do not introduce new verbose logging without toggle.

---

## 8. JSON / Profile Compatibility

If removing a setting:
- Ensure existing JSON profiles do not break.
- Prefer leaving legacy property unused rather than deleting unless explicitly approved.

---

## 9. Definition of Done

A task is only complete when:

- Code compiles.
- Scope respected.
- No unintended behavior changes.
- Documentation updated.
- `Docs/RepoStatus.md` updated with new validation commit.
- Relevant GitHub-facing docs reviewed and updated when the task changes anything user-facing.
- `CHANGELOG.md` updated when the task changes user-visible behaviour worth surfacing in release history.
