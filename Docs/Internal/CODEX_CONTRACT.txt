# LALALAUNCH CODEX CONTRACT

This contract applies to ALL Codex tasks unless explicitly overridden.

Codex must treat this as mandatory engineering policy.

---

## 1. Scope Discipline (No Feature Creep)

* Modify ONLY the files explicitly listed in the task, plus mandatory documentation files required to keep repo truth aligned.
* Do NOT expand scope.
* Do NOT introduce new subsystems.
* Do NOT add new UI controls unless explicitly requested.
* Do NOT change learning, shift logic, fuel math, CarSA logic, or telemetry gating unless explicitly requested.

If something appears wrong but is outside scope:

* Do not fix it.
* Do not refactor it.
* Leave it untouched.

---

## 2. Documentation Is Mandatory

Every code change must update relevant documentation.

### Technical / repo docs:

* Docs/Internal/SimHubParameterInventory.md
* Docs/Internal/SimHubLogMessages.md
* Docs/Subsystems/<affected_subsystem>.md
* Docs/Internal/Plugin_UI_Tooltips.md
* Docs/Project_Index.md
* Docs/RepoStatus.md

### GitHub-facing user docs (product surface):

* README.md
* CHANGELOG.md
* Docs/Quick_Start.md
* Docs/User_Guide.md
* Relevant feature pages (Dashboards, Strategy_System, H2H_System, etc.)

GitHub-facing documentation is part of the product.

If a change affects:

* user workflow
* UI behaviour
* feature meaning
* dashboard usage

Then user docs MUST be reviewed and updated.

Failure to update these when required = incomplete task.

Rules:

* Always classify changelog impact: user-facing / internal-only / both
* Update Development_Changelog.md for all non-trivial work
* Update root CHANGELOG.md only when preparing user-facing release notes
* Remove outdated documentation
* Do not document inactive systems as active

No task is complete without documentation alignment.

---

## 3. Codex Execution Method (Mandatory Working Order)

Before ANY non-trivial task:

* Read Docs/Project_Index.md
* Read Docs/Internal/CODEX_CONTRACT.txt
* Read relevant Docs/Subsystems/*.md
* Check Docs/RepoStatus.md when current state matters
* Read relevant entries in Docs/Internal/Development_Changelog.md

Treat Code_Snapshot.md as orientation only.

---

## 4. Required Analysis Before Editing

Codex MUST explicitly state:

* Owning subsystem
* Upstream dependencies
* Downstream consumers
* Relevant recent changes (from Development_Changelog.md)
* System invariants
* Edit boundaries

Codex MUST also declare:

* Does this change behaviour? (yes/no)
* If yes: exact impact

---

## 5. Stability First

* Do NOT alter runtime behavior outside scope
* Do NOT introduce timing side effects
* Do NOT block main thread
* Do NOT introduce race conditions

---

## 6. Regression Prevention (CRITICAL)

Codex MUST evaluate:

* What could this change break?
* Why is it safe?

If Codex cannot answer this clearly:

* Do not proceed

---

## 7. No Silent Contract Drift

Do NOT change any of the following without updating docs:

* exports
* log messages
* UI wording
* workflow semantics
* settings meaning

All changes must be reflected in canonical docs in the same task.

---

## 8. C# and Dependency Rules

* Target C# 7.3
* No new NuGet packages
* No external DLLs
* No unsafe code
* No async changes unless requested

---

## 9. Logging Discipline

* No log spam
* Use one-time warnings where possible
* Maintain existing log format

---

## 10. JSON / Profile Compatibility

* Do not break existing profiles
* Prefer legacy fields over deletion unless approved

---

## 11. Definition of Done

A task is complete ONLY when:

* Code compiles
* Scope respected
* No unintended behaviour changes
* Regression risks evaluated
* Documentation updated
* RepoStatus updated correctly
* GitHub docs reviewed
* Changelog classification included

## 12. Edge Case Awareness

Codex MUST consider and explicitly evaluate edge cases including:

- session transitions
- telemetry gaps or missing values
- boundary conditions (zero, max, empty, full)
- timing edges (start/end of lap, pit entry/exit, race end)

If edge cases are relevant to the change:
- they must be explicitly addressed in analysis and verification

If no edge cases apply:
- Codex must explicitly state why
